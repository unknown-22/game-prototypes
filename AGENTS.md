# instructions

**日本語で返答してください**

ゲームアイデアの生成とプロトタイプ作成を行うモノレポ。
prototypes/ 以下に 001_<プロトタイプ名>/main.py という構成でプロトタイプを配置してください。

- 言語: Python 3.12
- フレームワーク: pyxel

**プロジェクトルール**

- python 3.12のモダンな機能を活用する
- 型ヒントを積極的に使用する
- ruff(lint)とty(type checker)を活用してコード品質を保つ
- 適切な単位でモジュール化・クラス化を行う
- ロジック部分はクラス化・関数化してテスト可能にする
- linterや型チェックで関数・引数ミスが出たら inspect モジュールで関数シグネチャを確認するワンライナーを都度作成し、調査と修正を繰り返す
- 作成後に docs/prototypes.json を更新する

## ゲームプロトタイプ設計指針

プロトタイプでは完成度や物量よりも、「遊びの核」が短時間で伝わることを優先する。
プレイヤーが判断し、操作し、結果を受け取り、次はもっと上手くやろうと思えるループを必ず含める。

**ゲームとして面白くするための観点**

- プレイヤーの選択に意味を持たせる
  - 「リスクを取るか安全に進めるか」「どの順番で処理するか」「今リソースを使うか温存するか」のような判断を入れる
  - 良い選択は明確に得をし、悪い選択は納得できる損につながるようにする
- 短いプレイループを作る
  - 見る → 判断する → 操作する → 結果が返る → 次の判断へ、という流れを数秒から十数秒で回す
  - プロトタイプ中に同じ面白さが何度も発生するようにする
- 失敗理由をわかりやすくする
  - 敵の予兆、危険範囲、残り時間、HP、リソース不足などを画面上で伝える
  - 理不尽な失敗ではなく、次はどう改善できるかが見える失敗にする
- 成功を気持ちよく見せる
  - スコア加算、敵撃破、連鎖、回避成功、強化獲得などに即時フィードバックを入れる
  - Pyxelでは色変化、短いエフェクト、SE、数値ポップアップなどを活用する
- 緊張と緩和を入れる
  - 危険な局面、回復や強化のタイミング、ボーナスチャンスを交互に配置する
  - 30秒から1分のプレイでも展開が変化するようにする
- 上達できる余地を残す
  - ランダム性だけで結果が決まらないようにする
  - 操作精度、判断速度、ルート選択、リソース管理など、プレイヤーの腕前が出る要素を入れる

**プロトタイプに含めるべき要素**

- 明確な目的
  - 例: 60秒生き残る、スコア100を目指す、ボスを倒す、出口に到達する
- 失敗条件
  - HPが0、時間切れ、敵に捕まる、リソースが尽きるなど
- スコアまたは進行度
  - 前回より上手くなったか比較できる指標を入れる
  - 例: スコア、到達階層、生存時間、コンボ数
- リスクとリターン
  - 危険地帯に行くと高得点、強い攻撃は隙が大きい、アイテムを取るには敵に近づく必要がある、など判断が生まれる構造を入れる
- 段階的な難度上昇
  - 時間経過で敵が増える、ステージが狭くなる、敵の種類が増えるなど
  - 急に理不尽にせず、プレイヤーが変化を理解できるようにする
- 気持ちいい瞬間
  - コンボ、連鎖、クリティカル、回避直後の反撃、まとめて倒す、強化で一気に形勢逆転する場面などを最低1つ入れる
- 速いリスタート
  - ゲームオーバー後にすぐ再挑戦できるようにする
  - タイトル画面や凝ったメニューより、再プレイのテンポを優先する

**プロトタイプに必要な画面**

- 少なくとも以下の3画面を用意する
  - タイトル画面
  - ゲーム画面
  - ゲームオーバー画面
- タイトル画面では、ゲームタイトルと操作方法を表示する

**作り始める前に決めること**

まず「このゲームで一番面白い瞬間は何か」を1文で決める。

例:

- 敵をギリギリまで引きつけて一網打尽にするのが面白い
- 限られた手札から最善の合成を選ぶのが面白い
- 危険なルートほど報酬が高く、欲張るか迷うのが面白い
- 操作ミスしそうな速度の中で、リズムよく回避するのが面白い

プロトタイプでは、その1文を支える状況、ルール、成功/失敗のフィードバック、再挑戦したくなる指標を優先して実装する。
長いストーリー、多数のステージ、大量のアイテム、凝ったメニュー、複雑な成長システムは、遊びの核が確認できてから追加する。

## 質が高いプロトタイプの作成基準

質が高いプロトタイプとは、見た目や物量が多いものではなく、「仮説が明確で、短時間で遊びの良し悪しを判断でき、改善点が見えるもの」とする。
作成時は以下を満たすこと。

**実装前に決める仮説**

- 体験仮説
  - プレイヤーにどんな感情を起こしたいかを1文で書く
  - 例: 「危険な範囲に踏み込んでから脱出すると、怖いが得をした感じがする」
- メカニクス仮説
  - その感情を生むための操作、ルール、制約、報酬を明確にする
  - MDAの考え方に沿って、Mechanics(ルール)がDynamics(プレイ中の駆け引き)を生み、Aesthetics(感じてほしい面白さ)につながっているか確認する
- 検証方法
  - 1プレイ30秒から90秒で、仮説が当たっているか判断できる状態にする
  - 「何を見れば成功か」をスコア、到達時間、コンボ、被弾回数、選択回数などの観測可能な指標にする

**操作感とフィードバック**

- 入力に対する反応は即時に返す
  - 移動、攻撃、決定、キャンセルなど主要操作に遅延や曖昧さを残さない
  - 押した瞬間に変化が見えるよう、アニメーション開始、色変化、SE、エフェクトのいずれかを入れる
- 重要イベントには複数のフィードバックを重ねる
  - 命中、被弾、回避成功、コンボ成立、報酬獲得、危険予兆は、画面表示だけでなく音、点滅、短い停止、ノックバック、数値ポップアップなどを組み合わせる
  - ただし遊びの判断を邪魔する過剰な揺れ、長い演出、読めない点滅は避ける
- 危険と報酬は見分けやすくする
  - 危険範囲、次に起きる攻撃、取るべき報酬、現在のリソースを画面上で常に判断できるようにする
  - 色だけに依存せず、形、位置、点滅パターン、ゲージなども使う

**ゲームバランス**

- 初回プレイで目的と失敗理由が理解できる難度にする
  - 開始直後は安全に試せる猶予を作る
  - 難度上昇は、敵数、速度、攻撃間隔、地形制限、報酬条件のうち1から2要素ずつ変化させる
- リスクとリターンを数値で調整する
  - 危険な行動ほど、スコア、回復、コンボ倍率、時間延長など明確な得を返す
  - 安全行動だけで最適にならないよう、時間制限、得点効率、報酬の消滅などで判断を促す
- ランダム性は再挑戦を壊さない範囲にする
  - 運で詰む配置を避ける
  - ランダムな出現には予兆、間隔、上限、逃げ道を用意する

**実装スコープ**

- 最初に1つの完成したプレイループを作る
  - プレイヤー操作、敵または障害、報酬、失敗、スコア、リスタートまでを先に通す
  - 追加要素は、そのループを強くするものだけに絞る
- 1プロトタイプ内の主要メカニクスは1から2個に制限する
  - 例: 「チャージ攻撃」と「回避後ボーナス」まではよいが、クラフト、育成、装備、会話を同時に入れない
- 画面上の情報量を絞る
  - 常時見る必要がある情報は、HP、時間、スコア、リソース、次の危険のうち必要最小限にする
  - 説明文を増やすより、ゲーム中の配置とフィードバックで理解できるようにする

**テストと検証**

- ロジックはPyxel依存から分離する
  - スコア計算、当たり判定、敵の出現制御、難度上昇、リソース増減は関数またはクラスに切り出してテストする
  - 描画と入力だけをPyxel側に寄せる
- 最低限の自動テストを追加する
  - リスク行動の報酬、失敗条件、コンボや倍率、難度上昇、リスタート時の初期化をテスト対象にする
  - ランダムを使う場合はseedや乱数生成器を注入し、再現可能にする
- 手動確認では以下を必ず見る
  - タイトルからゲーム開始、ゲームオーバー、再スタートまで止まらず動く
  - 30秒以内に「面白い瞬間」が最低1回起きる
  - 失敗した理由が画面上の情報から説明できる
  - スコアや到達度が前回プレイと比較できる
  - ruff、ty、pytestが通る

**完成時に残す記録**

- docs/prototypes.json は既存スキーマに合わせて更新し、ideaやstatusの文面で「面白い瞬間」と「リスクとリターン」が伝わるようにする
- 必要に応じてプロトタイプのディレクトリに README.md を置き、以下を記録する
  - 体験仮説
  - 実装したコアループ
  - うまくいっている点
  - 次に改善するなら最初に触る点

## 開発環境とコマンド

### 基本コマンド

uvを利用します

```bash
# 実行
uv run python main.py

# pytest実行
uv run pytest tests/
uv run pytest tests/test_module.py

# Ruff実行
uv run ruff check
uv run ruff check <file_name> --fix

# 型チェック(ty)
uv run ty check
uv run ty check <file_name>

# app/webビルド、docs/への配置
uv run pyxel package prototypes/001_alchemy_deckbuilder prototypes/001_alchemy_deckbuilder/main.py
uv run pyxel app2html 001_alchemy_deckbuilder.pyxapp
mv 001_alchemy_deckbuilder.html docs/
```

### Pyxel Tips

Pyxel利用時に気をつけるべきことを完結に記載する

- 日本語を利用する場合は assets/k8x12.bdf をプロトタイプディレクトリ配下にコピーして、pyxelのフォントとして読み込む
  - k8x12.bdf はサイズが8x12のため、それに合わせて調整する
- BDFフォントなど外部アセットを使う場合、webビルドで `pyxel package` に含まれるようにプロトタイプディレクトリ配下へコピーして参照する
  - 例: `prototypes/054_mole_chain/k8x12.bdf` を `Path(__file__).with_name("k8x12.bdf")` で読み込む
- 絵文字や特殊文字は利用しない
- 解像度は 320x240 にする
- 操作にマウスを利用する場合、ゲームにあったマウスカーソルを用意する
